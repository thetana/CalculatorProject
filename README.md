[프로젝트 설명]

콘솔에서 사용자에게 입력받은 값으로 사칙 연산을 하고 계산 이력을 기록하는 CLI 프로그램 이다

[파일별 설명]

Main : 프로젝트의 시작점 이다 시작하면 App 오브젝트화 하고 실행한다.

App : 현재는 메인과 유사한 역할을 한다 이후에 확장 프로그램의 흐름을 총괄 하려고 만들었다

Page : 콘솔에서 이루어지는 입력과 출력 즉 사용자 인터페이스를 담당하는 클래스들을 이 프로젝트에서는 Page라고 하는데 그 Page들의 부모가 되는 추상 클래스 이다
template 이라는 폴더에 Page들은 모아 두었다 웹에서의 프론트앤드, 안드로이드의 액티비티 쯤의 포지션을 담당하고 있다

MainPage : 처음 유저가 보게 되는 Page이다 index.html 이나 메인 액티비티 이런거랑 비슷하다
이 MainPage에는 router라는 Map이 존재하는데 유저가 이후에 이동 할 수 있는 Page들을 등록해두면 자동으로 해당 페이지들을 보여주고 해당 페이지를 콘솔에서 선택 할 수 있게 해주는 역할을 한다

CalculationPage : 이 프로젝트의 핵심 기능인 계산하는 기능의 유저 인터페이스 이다 특이사항이 있다면 유저 입력을 한줄에서 처리 하도록 헀다 1 + 1 이런식으로 입력하면 된다

RecordPage : 계산된 이력을 보는 Page인데 진입하면 나름의 사용자 가이드가 있기는 하지만 사용법이 다소 어려울 수도 있다
 - 우선 지금까지 등록된 계산 이력을 보여준다
 - 계산 이력을 보고 특정 이력에 코멘트를 달고 싶다면 "c [키] [코멘트 내용]" 이렇게 입력 하면 되는데 예를 들면 "c 3 기록1" 이런식이다
 - 계산 이력중 삭제 하고 싶은게 있다면 "d [키]" 이렇게 입력하면 된다

ViewRecordsPage : 이전에 내 머리속에는 Page와 구분되는 View라는 개념이 있었다 입력은 받지 않고 조회만을 담당하는 Class 인데 CLI 환경에서 이걸 구분해서 클래스를 설계 하자니 실제 구현에서 코드가 좀 난해해 지는 느낌을 받아서 폐기하고 Page를 상속받아 출력만을 담당하는 페이지를 따로 만들었다
사실 RecordPage하나로 ViewRecordsPage가 하는 기능까지 수행해도 문제가 없지만 ViewRecordsPage의 기능은 계산된 이력을 조회하고 렌더링 하는 기능을 담당하는데 이게 따로 분리가 되어 있어야 향후에 다른 페이지들이 만들어졌을 때도 로직을 반복하지 않고 참조하여 사용이 가능하기 때문에 이렇게 만들었다
경험상 핵심 데이터의 조회는 빈번하게 일어난다 근데 다른 Page가 만들어진다 한들 거기선 조회 조건이 다르면 어떡하냐 라고 생각 할 수 있다 그걸 해결 하는 것이 바로 이 ViewRecordsPage에 내장된 filter라는 맵이다 저 filter 맵은 유저가 원하거나 아니면 개발자가 고정한 특정 필터 값을 기준으로 stream()을 활용하여 데이터를 원하는 형태로 볼 수 있도록 해준다

Calculateable : calculate라는 메소드를 선언한 인터페이스 이다 제네릭하게 만들려고 했고 실질적인 로직 구현은 ArithmeticCalculator에서 하고 있다

ArithmeticCalculator : Calculateable인터페이스를 구현하는 계산기 클래스 이다 계산의 연산자 구분은 OperatorType이라는 이넘을 활용 하고 있고 계산되는 데이터 타입은 BigDecimal을 활용한다 실수 까지 수용하면서 정확한 계산을 하기 위함이다

OperatorType : 연산자를 구분하는 역할을 한다 BiFunction을 적용해서 구분을 하고 적절한 계산까지 수행 할 수 있도록 했다

CalculationRecorder : 연산된 데이터의 CRUD를 수행하는 클래스 이다 향후에 JSON 파일이나 SQLite등을 이용해 영구저장을 만들면 저장소 클라이언트로써 동작 할 수 있게 로직을 설계했다

Calculation : "계산"이라는 개념 그 자체인 데이터 오브젝트 이다 특이한건 딱히 없는데 toString()을 오버라이드 하여 계산의 기본 출력 양식을 정의해서 print를 좀 편하게 하고 있다
